先分類錯誤

現象可能原因優先處理
找到完全不相關的文章段落 Retrieval 召回失敗 判斷是語意型或關鍵字型，再調 vector_weight / keyword_weight。
正確段落有出現但排名很後面 候選池或 rerank 不夠 先加 candidate_pool_size，再視情況加 rerank_weight。
正確段落有進結果但回答仍不準 Answer synthesis 問題 不要只靠 profile 解；需要看 prompt、context 或下一階段 judging。
分數很高但全 miss Ground truth 太窄、substring 不合理或 chunk 被切開 先修 Eval item，不要急著調參。
常常回答資料不足但其實有資料 Low-confidence threshold 太高 降低 low_confidence_threshold。
常常硬答或引用弱證據 Threshold 太低或 context 太雜 提高 low_confidence_threshold，或降低 final_chunk_count。

調參方向速查

問題類型建議方向注意事項
藥名、代號、條款、表格數字、檔名等精確查找 提高 keyword_weight，例如 0.3 到 0.4；vector_weight 對應降低。 檢查 exact-term 題改善時，語意題是否退步。
同義改寫、概念型問題、中文問英文文件 維持或提高 vector_weight，例如 0.7 到 0.75；keyword_weight 對應降低。 檢查藥名 / 數字型問題是否明顯變差。
正確段落常在 top-k 內但不是第一名 提高 candidate_pool_size 到 30 或 40；再試 rerank_weight 0.85 / 0.9。 候選池越大，成本與延遲可能上升。
回答引用太多雜訊 降低 final_chunk_count，例如 6 到 4；或提高 rerank_weight。 更多 context 不等於更準，可能讓答案漂移。
拒答太多 降低 low_confidence_threshold，例如 0.25 到 0.2。 檢查 hallucination 和弱引用是否增加。
亂答太多 提高 low_confidence_threshold，例如 0.25 到 0.3。 檢查本來可回答的題目是否被擋掉。

建議實驗流程

  1. 固定一組 Eval Set，包含客戶實際抱怨的不準案例，並覆蓋精確查找、語意改寫、跨語言、相似文件辨識、無答案問題。
  2. 先跑 baseline profile，不先改任何參數。
  3. 逐題看 miss diagnosis、top retrieved chunk、expected evidence，人工分類錯誤。
  4. 建立 2 到 3 個 candidate profile，例如 keyword-heavy、semantic-heavy、rerank-heavy。
  5. 每次只改一個方向，跑同一組 Eval Set，比較 Recall、MRR、miss 題目和原本正確題目是否退步。
  6. 只有在整體錯誤分布改善，且沒有明顯讓原本正確的題目退步時，才套用到 live retrieval。

起手 Profile 範例

keyword-heavy

客戶問題偏藥名、代碼、條款、表格數字。

keyword_weight 0.3 到 0.4
vector_weight 0.7 到 0.6

semantic-heavy

客戶問題偏同義改寫、概念查詢、跨語言。

vector_weight 0.7 到 0.75
keyword_weight 0.3 到 0.25

rerank-heavy

正確段落有被找回但排序不佳。

candidate_pool_size 20 到 30/40
rerank_weight 0.8 到 0.85/0.9

strict-evidence

系統常硬答或引用弱證據。

low_confidence_threshold 0.25 到 0.3
final_chunk_count 6 到 4

領域提示與回答規則

Notebook 內的 domain hints 有機會提高準確度，尤其是領域詞彙對齊和回答規則一致性；但它不是替代索引或 ground truth 的萬靈丹。

適合放入 hints

  • 俗稱、縮寫、內部代號、產品別名
  • 跨語言術語與常見問法
  • 固定回答規則，例如數字必須來自引用依據

避免的用法

  • 把一大段 domain know-how 當成萬用 system prompt
  • 把 prompt 當成未引用的知識來源
  • 未經 Eval 比較就直接上線
