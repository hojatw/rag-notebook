# I18N.md — UI internationalisation reference

How UI copy is localised, how to add strings, how to add/switch locales, and the
planned admin + per-user locale controls. The implementation is the **U15a
foundation** in [`ROADMAP.md`](ROADMAP.md); the per-user/admin controls are the
remaining **U15b** work described at the end.

Scope: this is **UI display copy only**. LLM prompt language (what the model
writes in answers/briefings/etc.) is a separate concern and must NOT be routed
through this catalog — see [`RETRIEVAL.md`](RETRIEVAL.md) / the prompt constants
in `app/llm.py`.

## Architecture

Dependency-free dict catalog — no gettext, no build step (keeps the POC's
"no npm / no build" constraint).

| Piece | Where | Role |
|---|---|---|
| `MESSAGES` | [`app/i18n.py`](../app/i18n.py) | `{locale: {dotted.key: text}}`. `zh-TW` is the source locale and the only fully-populated one. |
| `DEFAULT_LOCALE` | `app/i18n.py` | `"zh-TW"` — the fallback locale; every key must exist here. |
| `active_locale()` | `app/i18n.py` | Returns the locale to render, read from `config.ui.language` **at call time**. |
| `t(key, **kwargs)` | `app/i18n.py`, registered as a Jinja global in `app/main.py` | Template lookup: `{{ t("nav.notebooks") }}`. `**kwargs` fill `{name}` placeholders. |
| `js_messages()` | `app/i18n.py` | Returns the `js.*` keys (prefix stripped) for the active locale. |
| `window.I18N` | emitted in [`app/templates/base.html`](../app/templates/base.html) head | `<script>window.I18N = {{ i18n_js() | tojson }}</script>` — runs before the deferred `app.js`. |
| `tr(key, fallback, vars)` | [`app/static/app.js`](../app/static/app.js) | Client-side lookup against `window.I18N`; `vars` fill `{name}` placeholders. |
| `[ui].language` | [`app/config.py`](../app/config.py) `UIConfig`, [`config.example.toml`](../config.example.toml) | The configured locale. Env override: `NOTEBOOKLM_UI_LANGUAGE`. Default `zh-TW`. |

**Fallback chain** (in `t()` and `js_messages()`): active locale → `DEFAULT_LOCALE`
→ the key string itself. A missing translation therefore degrades visibly-but-safely
(you see the source-locale text, or the raw key) instead of raising.

**Key naming**: dotted, grouped by surface — `nav.*`, `common.*`, `chat.*`,
`studio.*`, `tool.*`, `eval.*`, `prof.*`, `evalr.*`, `audit.*`, `error.*`,
`flow.*`, `export.*`, `a11y.*`, and `js.*` (the only group emitted to the client).

### Status label maps (the exception to the catalog)

Lifecycle/classification values come from the DB as raw identifiers
(`indexed`, `succeeded`, `hit`, `high`, …) and are localised through plain dicts,
not `t()`, via `.get(value, value)` so the stored identifier is preserved:

| Map | Location | Used for |
|---|---|---|
| `source_status_labels` | inline dict in [`app/main.py`](../app/main.py) | source lifecycle (已索引/處理中/失敗/已上傳) |
| `SENSITIVITY_LABELS` | `app/i18n.py` | audit severity (高/一般/低) |
| `RUN_STATUS_LABELS` | `app/i18n.py` | eval run status (排隊中/執行中/成功/失敗) |
| `EVAL_RESULT_STATUS_LABELS` | `app/i18n.py` | per-question result (命中/未命中/錯誤/未評分) |

> ⚠️ **These maps are single-locale (zh-TW only).** They do **not** switch with
> `active_locale()`. Making them locale-aware is a required step for any second
> locale (see "Add a new locale" below).

## Add a new UI string

Never hardcode user-facing copy in a template or in `app.js`. Instead:

1. Add the key to `MESSAGES["zh-TW"]` in `app/i18n.py`, in the right group.
2. **Template**: `{{ t("group.key") }}` (or `{{ t("group.key", name=value) }}`
   for placeholders).
3. **Client JS**: put the key under `js.*` and read it with
   `tr("key_without_prefix", "zh fallback", { name: value })`. It's auto-emitted
   into `window.I18N`.
4. **Server (`main.py`)**: `i18n.t("group.key")`.
5. A new status *value* (not a label) goes in the relevant status label map, not
   the catalog.

## Change the deployment locale

Deployment-wide, all users, single setting:

- `config.toml`: `[ui]\nlanguage = "en"`, **or**
- env: `NOTEBOOKLM_UI_LANGUAGE=en`

Caveats:
- ⚠️ Config is loaded **once at process start** (`config.py` runs `load_config()`
  at import) — **restart** to apply.
- ⚠️ Setting a locale that isn't populated in `MESSAGES` shows the zh-TW
  fallback for every key, i.e. **no visible change** until that locale's catalog
  exists.
- ⚠️ This is **not** per-user (see U15b below).

## Add a new locale (e.g. `en`)

**Minimum (catalog-routed copy switches):**
1. Add `"en": { …same keys… }` to `MESSAGES` in `app/i18n.py`. Missing keys
   fall back to zh-TW, so it can be filled incrementally.
2. Activate via `NOTEBOOKLM_UI_LANGUAGE=en` (+ restart). `window.I18N` picks up
   the locale's `js.*` automatically.

**For full coverage, also:**
3. Make the four status label maps locale-aware — restructure to
   `{locale: {...}}` (or fold them into the catalog behind a small helper).
   Otherwise status pills stay in zh-TW after switching.
4. Extract the documented inline exceptions (below) if they must translate.
5. Add a test asserting **key parity** between `en` and `zh-TW` (today the test
   only checks the zh-TW source set — see [`tests/test_i18n.py`](../tests/test_i18n.py)).

## Known exceptions (deliberate, not omissions)

- **Deepest already-zh eval prose** (the long explanatory sentences in
  `admin_eval_set.html` / `admin_eval_help.html`) is left as inline literals,
  not catalog-routed. Display language is unaffected; this avoids large
  zero-visible-change churn. Extract it only when a second locale needs it.
- **`PROFILE_PARAM_LABELS`** (`app/retrieval.py`) — technical retrieval-parameter
  names — are preserved per [`UI.md`](UI.md) §5.
- **Domain terms** kept verbatim everywhere: Recall, MRR, Profile, chunk,
  embedding, RAG, and the eval data nouns `Eval Set` / `Eval run` / `Run` /
  `Baseline` / `Candidate`. Eval *activity/section* chrome is Chinese (`評測`).

## Planned: admin system locale + per-user locale (U15b)

Target behaviour:
- **Admin** sets the **system default locale** from the UI (System settings),
  persisted (not just env/file), applied to users who haven't chosen their own.
- **Each user** can override with a **personal locale** preference, persisted
  per account, applied to their own session.

Design notes / what this requires beyond today's foundation:
1. **Persistence**: a system-locale setting (e.g. in `llm_settings`/a settings
   table) + a per-user `locale` column on `users` (schema change → update
   [`SCHEMA.md`](SCHEMA.md)). Resolution order: user locale → system locale →
   `DEFAULT_LOCALE`.
2. **Request-scoped resolution**: today `t()` reads the global `config.ui.language`.
   Per-user means resolving the locale **per request** (e.g. a `contextvar` set
   from the signed-in user, or threading locale through render context) so `t()`
   and `js_messages()` use the request's locale, not a process-global.
3. **Locale-aware status maps** (step 3 above) become mandatory.
4. **UI affordances**: a language control in System settings (admin, system
   default) and in the account page (per-user).
5. **Tests** across HTMX partials (each partial must localise under the
   request's locale, including fragments fetched after navigation).
6. Populate the actual `en` catalog (the foundation only ships `zh-TW`).

Keep LLM prompt-language rules separate from UI i18n throughout.
